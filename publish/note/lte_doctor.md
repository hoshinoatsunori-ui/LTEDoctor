# LTEモジュールが突然切れる。原因がわからない。その地獄を終わらせた話

IoT製品の開発をしていると、ある日突然こんな報告が来る。

「現場でLTEが切れた。でも再現しない」

原因を調べようとすると、ログが4種類ある。しかも時刻がバラバラだ。ATコマンドのログはWindowsの時計（日本時間）、モジュール内部のデバッグログはチップが起動してからの経過マイクロ秒、Wiresharkのパケットキャプチャは世界標準時、消費電流の波形ログは測定開始からの相対時間。

これを手動でエクセルに貼って「だいたいこの辺」と目視で突合する作業が始まる。1件の解析に半日かかる。そしてまた現場から報告が来る。

この地獄を終わらせるPythonツール「LTE Doctor」を作った話をする。


## 解決策：4種のログを「時刻」で統合する

考え方はシンプルだ。バラバラな時計を一本の時間軸に揃えて、全部のログを重ねて見る。

具体的には「アンカー」と呼ばれる基準点を使う。たとえばモジュールが「初期化完了」のメッセージを出した瞬間は、ATコマンドログにもデバッグログにも同時に記録される。この「同じ出来事が複数のログに記録された瞬間」を基準点にして、各ログの時刻をずらして揃える。

基準点が複数あれば、間を補間してさらに精度を上げる。時計のドリフト（少しずつずれていく現象）にも対応できる。

実装上は、Wiresharkのパケットキャプチャを「最も信頼できる時計」として使う。GPS同期されたUTC時刻が入っているからだ。ATコマンドログはPCの時計で補正、チップセット内部のマイクロ秒カウンタはブート時刻から逆算してUTCに変換する。

4種のログが揃ったら、SQLiteデータベースに入れて診断ルールにかける。ルールは「接続してから2分以内に切断」「リセットが30秒以内に3回」といった条件をYAMLファイルで定義してある。合致したものを重大度（HIGH/MEDIUM/LOW）付きで出力する。


## 実際に発見した障害パターン

2026年4月13日のログで試してみた。

午後5時9分、モジュールが起動した。そこから4分間、Band1（周波数帯の一種）での接続を試み続けて、9回連続でリセットを繰り返した。電波はつかまえているが、ネットワークへの登録が通らない。

午後5時12分35秒、手動でBand40に切り替えるATコマンドを送った。

24秒後、セルを捕捉した。さらに1分後の午後5時13分57秒、ついに接続成功。電波品質も-109dBmから-97dBmに改善していた。

しかし57秒後、ネットワーク側から切断を告げるメッセージが届いた。「PDN接続エラー、原因コード30」。原因コード30はネットワーク側の拒否を意味する。Wiresharkのパケットを見ると、ネットワーク側からDetachリクエストが来ていることが確認できた。

LTE Doctorが自動で出した診断：

- **R-006 HIGH**：リセットカスケード（4分で9回）
- **R-001 HIGH**：ネットワーク起因の切断
- **R-003 HIGH**：PDN接続エラー、原因コード30
- **R-004 MEDIUM**：接続から2分以内の短命セッション

これらが `python run_pipeline.py` を1回実行するだけで出てくる。


## ツールはどうやって使うの？

使い方は3ステップだ。

**Step 1** 消費電流データをCSVに変換する

```
python otii2csv.py
```

OTII Arcという測定器から出てくるデータを標準的なCSV形式に変換する。

**Step 2** ログを全部まとめて解析する

```
python run_pipeline.py
```

4種のログを読んで、時刻を揃えて、診断して、結果をデータベースに保存する。

**Step 3** Webブラウザで結果を見る

```
python doctor.py
```

`http://localhost:5001` を開くと、こんな画面が出る。

```
┌─────────────────────────────────────────────┐
│  LTE Doctor  [Setup] [Sessions] [Timeline]  │
├─────────────────────────────────────────────┤
│ 消費電流波形（Plotly グラフ）               │
│  ↑↑↑ここでLTEセッション中のmA変動が見える  │
├─────────────────────────────────────────────┤
│ イベントマーカー（接続・切断・リセット等）  │
├──────────────┬──────────────┬───────────────┤
│ ルール詳細   │ PCAP一覧     │ ATコマンド一覧│
│ R-001: HIGH  │ Detach req   │ +KCNX_IND:1,5│
│ NW起因切断   │ 17:14:51     │ 17:14:50     │

```

グラフはズームやパンができる。電流波形でピークが上がった瞬間にどのLTEイベントが起きていたか、マウス操作だけで確認できる。


## 作ってみてわかったこと

**JSTとUTCの混在は本当に痛い**

WindowsのシリアルモニタはPCの時計で記録するから、ATコマンドログは日本時間（JST）になる。WiresharkはUTCで記録する。この9時間の差を見落とすと相関分析が全部狂う。モジュール内部の時計（`+CCLK`コマンドで確認できる）は別途約1分の遅れがあることも実測でわかった。時刻の扱いは思っている以上に厄介だ。

**消費電流波形とLTEイベントは完璧に対応する**

OTII Arcで計測した電流波形を見ると、LTEの接続フェーズ・認証フェーズ・データ送受信・切断のそれぞれで電流プロファイルが変わる。Wiresharkのパケットと重ねると、どのNASメッセージのやり取りでどれだけ電力を使っているかが一目瞭然だった。バッテリー駆動のIoT機器では電力最適化の強力な手がかりになる。

**57秒の接続はデバイス側の問題ではなかった**

Band40に切り替えて電波品質も改善していたのに57秒で切断された原因は、デバイスのファームウェアやアプリケーションではなく、ネットワーク側のPDNゲートウェイ設定かAPN認証の問題だと判断できた。ツールがなければ「なんとなく電波が悪かったのでは」という結論で終わっていたかもしれない。ログを統合することで、「どこの問題か」の切り分けが根拠を持ってできるようになった。


## 今後やりたいこと

通信先サーバー側のWiresharkとも突合できれば、往復遅延やTCPの再送状況まで見える。現状はモジュール側のキャプチャのみだ。

診断ルールを機械学習で自動生成する仕組みも面白そうだ。過去の障害ログを学習データにして、「この電流パターンの後にはリセットが起きやすい」といった予兆検知ができれば、フィールドトラブルの前に手を打てる。

シリアルポートから直接ログを取り込んでリアルタイムで診断するライブモードも作りたい。そうなれば「ログを持ち帰って解析」ではなく「現場で即座に原因特定」ができるようになる。

LTEモジュールの通信障害は「原因不明」で終わることが多かった。ログを統合して時刻を揃えて可視化するだけで、これほど明確に原因が見えるようになるとは思っていなかった。同じ課題で困っているIoTエンジニアの参考になれば嬉しい。
